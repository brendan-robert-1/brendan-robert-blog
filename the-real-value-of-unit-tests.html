The Real Value of Software Testing

In the ever-evolving world of software development, there are practices and concepts that often remain misunderstood. From the intricacies of "private" variables to the buzz surrounding agile methodologies, one concept that deserves our attention is Software Testing.

The Misunderstood Practices:

Software testing is typically categorized into various types, including unit testing, integration testing, and end-to-end testing. While many understand testing as a way to ensure software functions as expected, its true value goes beyond that. In this article, we'll delve into the multifaceted world of software testing, exploring its purpose, its significance for future-proofing your code, and a real-world example that highlights the importance of unit testing.

Types of Software Testing:

Software testing can be split into two or three main categories. "Unit" testing verifies the validity of small independent pieces of code, such as individual methods or classes, with everything else mocked out. "Integration" testing involves testing multiple independent pieces of code together, often involving method calls down through a stack of dependency-injected class instances. While integration testing usually mocks external dependencies like databases or external APIs, "End-to-End" testing is typically combined with integration testing but involves as little mocking as possible. It is not included in build pipelines to prevent breaking builds due to external factors like a database being down.

The Real Value of Unit Testing:

Many people assume that the purpose of software testing is merely to verify that the software performs as expected. While this is an important aspect, it's not the whole picture. Let's consider a scenario where you're tasked with writing a piece of business logic that behaves differently based on the month of the year. In the first half of the year, it makes an API call to one service, and in the second half, it calls a separate API. You may write unit tests to ensure the correct APIs are called based on the mocked month. You show these tests to your tech lead as proof that your code does what it says it does. Is this the end of the story?

The Importance Beyond Verification:

So, what's the crucial benefit of unit testing? It's about future changes. Allow me to share a brief story from a project I worked on. I joined a team that was tackling a legacy codebase approaching 1 million lines of code, and it had ZERO unit tests. Surprisingly, the code worked perfectly, doing exactly what it was supposed to do, and the business was thriving. Writing unit tests would have delayed the software and caused missed deadlines, so why bother?

As part of my initial contribution to this codebase, I fixed a simple bug related to date serialization in an endpoint. I deployed the fix but was asked to immediately revert it. Why? It turns out there were other processes that relied on the bug being present, while others needed the "fixed" date format. With ample unit testing, my change would have broken the methods that relied on the bug, revealing the issue at build time, not deploy time.

Finding the Right Balance:

Certainly, there are diminishing returns for time spent on unit testing. I've been on teams that aimed for 100% unit test coverage, leading us to test Plain Old Java Objects (POJOs) and getters/setters, which isn't ideal. A better metric to consider is how comfortable new developers feel deploying simple changes to production without breaking existing functionality.

Conclusion:

In the world of software development, software testing isn't just about ensuring that your code functions correctly today. It's about future-proofing your codebase and avoiding costly surprises down the road. While there's a balance to strike in testing efforts, finding that sweet spot can make all the difference in the stability and maintainability of your software.
